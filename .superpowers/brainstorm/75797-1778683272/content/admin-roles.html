Admin Interface — Who does what?

Clarifying the two roles before we design the interface

The current access control model has Akvo staff doing everything in the admin layer. The user wants to distinguish between Admin and Reviewer. Here's how those could split:

A

Admin = Akvo, Reviewer = CSO staff

Admin: Full access — configure sites, manage users, approve final publications, view all data
Reviewer (CSO): Flag, comment, and validate submissions for their assigned sites — but cannot publish or approve

B

Admin = Akvo, Reviewer = Akvo junior staff

Admin: Full access — configure, manage, publish
Reviewer: Internal Akvo role — can clean, tag, and score submissions but cannot approve for publication. Final approval stays with Admin

C

Three-tier: Admin / Akvo Reviewer / CSO Reviewer

Admin (Akvo): Configure, manage, publish
Akvo Reviewer: Clean, tag, score, approve
CSO Reviewer: Flag anomalies, add field notes — limited to their sites only
